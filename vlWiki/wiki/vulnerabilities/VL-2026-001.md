# SQL 注入

## 漏洞描述
未对用户输入正确执行危险字符清理

## 修复建议
查看危险字符注入的可能解决方案
常规
使用包含参数的存储过程，以防止在数据中注入 SQL 命令，或者至少使用不允许注入代码的参数化数据库调用。请勿在存储过程中包含执行动态 SQL 的任何语句。
更好的解决方案是使用 ORM（对象关系映射）框架，例如 Hibernate 或 EntityFramework（如果平台上提供了此类框架）。
确保在服务器端验证和过滤所有用户输入，这样不仅仅可以禁止使用诸如单引号 ( ' ) 和双引号 (") 之类的不良字符，而且只允许使用安全字符。根据请求中参数的期望值严密地定义安全
字符集。
对所有用户输入使用转义函数。
配置应用程序身份，以授予完成必要任务所需的最小数据库特权。加固数据库服务器以禁用任何不需要的功能，例如 shell 命令。

### 风险
可能的后果包括以下方面的损害：
机密性 - 由于 SQL 数据库通常用于存储敏感数据，因此丧失机密性是 SQL 注入漏洞的常见问题。
身份验证 - 如果使用不良的 SQL 命令检查用户名和密码，则有可能以其他用户的身份连接到系统，而无需提前知道密码。
授权 - 如果授权信息保存在 SQL 数据库中，则可以通过成功利用 SQL 注入漏洞来更改这些信息。
完整性 - 正如有可能读取敏感信息一样，通过 SQL 注入攻击还可以更改甚至删除这些信息。
